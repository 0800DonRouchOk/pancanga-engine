# Resumen

Este documento define el sistema temporal utilizado por Pancanga Engine.

Todo cálculo astronómico realizado por el motor dependerá de estas definiciones.

La representación del tiempo constituye uno de los pilares fundamentales del proyecto y deberá permanecer estable durante toda la vida útil del motor.

# 1. Objetivo

Definir un modelo temporal único para Pancanga Engine.

Todos los módulos del sistema deberán utilizar este modelo.

Ningún módulo podrá implementar su propio sistema de fechas u horas.

---

# 2. Principios

El sistema temporal deberá cumplir los siguientes principios.

## Determinista

La misma fecha siempre representa el mismo instante.

---

## Independiente del sistema operativo

El motor nunca dependerá del reloj del sistema para realizar cálculos internos.

El reloj del sistema únicamente podrá utilizarse para obtener la fecha actual.

---

## Independiente de Internet

No se consultarán servidores NTP ni servicios externos.

---

## Preciso

La precisión deberá ser suficiente para reproducir correctamente los cálculos del calendario Gauḍīya.

---

# 3. Escalas de Tiempo

Pancanga Engine reconoce las siguientes escalas.

## UTC

Tiempo civil internacional.

Será utilizado únicamente como interfaz con el usuario.

---

## UT1

Representa el tiempo basado en la rotación terrestre.

Será soportado por la arquitectura.

No será utilizado inicialmente.

---

## TT (Terrestrial Time)

Será la escala utilizada por los algoritmos astronómicos cuando corresponda.

---

## TDB

Queda reservado para futuras versiones.

No forma parte del alcance de la versión 1.

---

# 4. Representación Interna

Internamente el motor trabajará con un tipo denominado:

```rust
Instant
```

Este tipo representa un instante absoluto.

No representa una fecha.

No representa una hora.

Representa un punto del tiempo.

---

# 5. Tiempo Civil

Las fechas visibles para el usuario estarán representadas mediante:

```rust
CivilDate

CivilTime

CivilDateTime
```

Estos tipos existen únicamente para la interacción con el usuario.

Los cálculos internos utilizarán `Instant`.

---

# 6. Tiempo Astronómico

Todos los cálculos astronómicos utilizarán:

- Julian Day
- Escala temporal apropiada (TT cuando corresponda)

Los algoritmos nunca operarán directamente sobre fechas civiles.

---

# 7. Julian Day

Julian Day será la representación temporal utilizada por el motor astronómico.

Su implementación se define en PES-0003.

---

# 8. Precisión

La precisión mínima del sistema será la de `f64`.

En futuras versiones podrán incorporarse tipos de mayor precisión si fuese necesario.

Todas las operaciones deberán preservar la máxima precisión disponible.

---

# 9. Calendarios

La API pública aceptará fechas del calendario gregoriano.

Las conversiones al sistema interno serán responsabilidad del módulo temporal.

La gestión de fechas anteriores a la reforma gregoriana será definida en PES-0003.

---

# 10. Casos Especiales

El sistema deberá contemplar:

- años bisiestos;
- zonas horarias;
- horario de verano;
- años negativos (si el alcance del motor los incorpora);
- fechas históricas.

La implementación detallada se documentará en módulos posteriores.

---

# 11. API Pública

Los tipos públicos previstos son:

```rust
TimeScale

Instant

CivilDate

CivilTime

CivilDateTime

DeltaT
```

Estos tipos forman parte del contrato estable del motor.

---

# 12. Compatibilidad

El sistema temporal deberá permitir reproducir los resultados del calendario de referencia utilizado por Pancanga Engine.

Las diferencias entre escalas temporales deberán quedar documentadas y aisladas del resto del sistema.

---

# 13. Referencias

- Jean Meeus — Astronomical Algorithms (2nd Edition)
- IAU Standards
- SOFA (Standards of Fundamental Astronomy)
- Gaurabda Calendar (como referencia de compatibilidad)
- Documentación técnica del proyecto PureBhakti

---

# Criterios de Aprobación

PES-0002 se considerará aprobado cuando:

- exista una implementación de los tipos temporales;
- el sistema temporal sea independiente del sistema operativo;
- la API pública esté definida;
- todos los módulos posteriores utilicen exclusivamente este modelo.

---

# Nota de Revisión Técnica

A partir de PES-0002, las especificaciones documentan decisiones matemáticas y deberán contrastarse con estándares de astronomía como IAU, SOFA y Meeus antes de ser aprobadas.

# 1. Objetivo

Definir la arquitectura del subsistema astronómico que utilizará Pancanga Engine.

Este subsistema será responsable de calcular las posiciones de los cuerpos celestes y las magnitudes astronómicas necesarias para el calendario Gauḍīya.

No contendrá ninguna regla religiosa ni calendárica.

---

# 2. Principio Fundamental

La astronomía **no sabe** qué es una Tithi.

La astronomía **no sabe** qué es Ekādaśī.

La astronomía únicamente responde preguntas como:

> ¿Dónde está el Sol?

> ¿Dónde está la Luna?

---

# 3. Capas

```text
Astronomy
```

Cada módulo resuelve un único problema.

# 4. Coordenadas

Todo el sistema utilizará tipos explícitos.

## Coordenadas Eclípticas

```rust
pub struct EclipticCoordinates {
    pub longitude: Degrees,
    pub latitude: Degrees,
    pub radius_vector: f64,
}
```

- `longitude`: longitud eclíptica.
- `latitude`: latitud eclíptica.
- `radius_vector`: distancia al cuerpo (en unidades astronómicas para el Sol, otra unidad documentada para la Luna).

---

## Coordenadas Ecuatoriales

```rust
pub struct EquatorialCoordinates {
    pub right_ascension: Degrees,
    pub declination: Degrees,
    pub distance: f64,
}
```

---

## Coordenadas Horizontales

```rust
pub struct HorizontalCoordinates {
    pub azimuth: Degrees,
    pub altitude: Degrees,
}
```

---

# 5. Época de Referencia

Todo cálculo se realizará para un `Instant`.

La conversión desde fechas civiles ocurre antes de ingresar al subsistema astronómico.

---

# 6. Trait Principal

Todos los cuerpos celestes implementarán la misma interfaz.

```rust
pub trait CelestialBody {
    fn ecliptic_position(
        &self,
        instant: Instant,
    ) -> EclipticCoordinates;
}
```

En el futuro podrán agregarse métodos como posiciones aparentes o ecuatoriales sin romper la API.

---

# 7. Transformaciones

Existirá un módulo dedicado exclusivamente a convertir entre sistemas de coordenadas.

Ejemplos:

- Eclípticas → Ecuatoriales.
- Ecuatoriales → Horizontales.

Los cuerpos celestes **no** realizarán estas conversiones internamente.

---

# 8. Sol

Responsabilidad:

Calcular la posición eclíptica del Sol.

No calcula amaneceres.

No calcula estaciones.

No calcula Tithis.

---

# 9. Luna

Responsabilidad:

Calcular la posición eclíptica de la Luna.

No calcula Nakṣatras.

No calcula fases.

No calcula Tithis.

---

# 10. Nutación

Responsabilidad:

Calcular únicamente la nutación terrestre.

---

# 11. Oblicuidad

Responsabilidad:

Calcular la oblicuidad de la eclíptica.

---

# 12. Tiempo Sidéreo

Responsabilidad:

Calcular el tiempo sidéreo aparente y medio cuando sea necesario.

---

# 13. Ayanāṁśa

Responsabilidad:

Aplicar el modelo seleccionado (por defecto Lahiri).

La arquitectura permitirá incorporar otros modelos sin modificar el resto del sistema.

---

# 14. Transformaciones

Todos los cambios de sistema de coordenadas estarán centralizados en un único módulo.

Ejemplos:

```rust
pub fn ecliptic_to_equatorial(
    coords: EclipticCoordinates,
    obliquity: Degrees,
) -> EquatorialCoordinates;
```

---

# 15. Principios de Diseño

- Ningún módulo conoce las reglas Gauḍīya.
- Ningún módulo depende del Sol para calcular la Luna.
- Los cuerpos celestes no conocen la representación de fechas civiles.
- Todas las entradas usan tipos fuertes.
- Todas las salidas usan estructuras explícitas.

---

# 16. Testing

Cada módulo tendrá:

- pruebas unitarias;
- validación con ejemplos publicados;
- comparación contra referencias (Meeus, SOFA o equivalentes cuando aplique);
- pruebas de regresión.

---

# 17. Extensibilidad

La arquitectura permitirá incorporar:

- nuevos modelos solares;
- nuevos modelos lunares;
- otros cuerpos celestes;
- distintos ayanāṁśas.

Sin modificar la API pública.

---

# 18. Decisiones abiertas

Antes de implementar el Sol debemos resolver:

1. ¿Cómo representaremos las distancias? (`f64` simple o tipos fuertes).
2. ¿Las coordenadas serán siempre aparentes o distinguiremos entre geométricas y aparentes?
3. ¿Cómo modelaremos la época de referencia (J2000, fecha del cálculo u otras)?
4. ¿Qué precisión mínima exigiremos para considerar válido un algoritmo?

Estas preguntas deberán resolverse mediante ADR antes de comenzar la implementación del modelo solar.

---

# 19. Propuesta de Separación en Dos Capas

Hasta ahora el proyecto se ha construido como un motor compatible con PureBhakti.

ARCH-0002 propone revisar una separación explícita:

```text
Pancanga Engine

│

├── Astronomy Engine

└── Gaudiya Calendar Engine
```

- **Astronomy Engine**: completamente genérico, capaz de calcular posiciones del Sol, la Luna, coordenadas y transformaciones.
- **Gaudiya Calendar Engine**: utiliza ese motor para calcular Tithis, Nakṣatras, Ekādaśī, Mahādvādaśī y Pāraṇa.

Esta separación podría hacer que Pancanga Engine sea reutilizable para otros calendarios o aplicaciones astronómicas educativas.

Esta propuesta permanece en revisión y deberá evaluarse antes de iniciar la implementación del módulo del Sol.

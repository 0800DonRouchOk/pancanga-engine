# 1. Objetivo

Calcular la posición heliocéntrica de la Tierra usando `VSOP87D`.

Aunque el objetivo final de la cadena solar sea obtener la longitud aparente del
Sol vista desde la Tierra, el cálculo moderno comienza con la posición de la
Tierra alrededor del Sol.

La posición solar geocéntrica se obtendrá posteriormente invirtiendo la posición
heliocéntrica de la Tierra.

---

# 2. Cadena Astronómica

```text
Julian Date
    ↓
VSOP87 tau
    ↓
VSOP87D (Tierra)
    ↓
Longitud aparente del Sol
    ↓
Posición de la Luna
    ↓
Tithi
    ↓
Ekādaśī
```

---

# 3. Entradas

La entrada pública del algoritmo es:

```text
JulianDate
```

El Julian Day debe estar expresado en la escala dinámica compatible con VSOP87,
normalmente TDB para trabajos de alta precisión.

Internamente VSOP87D usa:

```text
tau = (JD - 2451545.0) / 365250.0
```

`tau` representa miles de años julianos desde J2000.0.

Nota: este algoritmo no usa `AST-0001` directamente porque `AST-0001` calcula
siglos julianos, mientras que VSOP87 usa milenios julianos.

---

# 4. Salidas

La salida implementada en esta etapa es la longitud heliocéntrica de la Tierra
en coordenadas esféricas referidas a la eclíptica y equinoccio de la fecha.

API actual:

```rust
pub fn earth_heliocentric_longitude(jd: JulianDate) -> Degrees;
```

La latitud heliocéntrica y el radio vector quedan fuera de esta primera entrega
de `AST-0002`.

---

# 5. Fuente Científica

Fuente primaria:

- VSOP87D.
- IMCCE / Observatoire de Paris, Planetary Solution VSOP87.

Fuente secundaria de implementación y explicación:

- Jean Meeus, Astronomical Algorithms, 2nd Edition.

`ADR-0004` adopta `VSOP87D` como variante oficial para la cadena solar inicial.
Esta entrega implementa el conjunto completo de términos de longitud
heliocéntrica terrestre publicados en `VSOP87D.ear`.

## Selección de Variante VSOP87

Según la documentación de IMCCE, las variantes relevantes incluyen:

- `VSOP87`: variables elípticas heliocéntricas, eclíptica y equinoccio J2000.
- `VSOP87A`: variables rectangulares heliocéntricas, eclíptica y equinoccio J2000.
- `VSOP87B`: variables esféricas heliocéntricas, eclíptica y equinoccio J2000.
- `VSOP87C`: variables rectangulares heliocéntricas, eclíptica y equinoccio de la fecha.
- `VSOP87D`: variables esféricas heliocéntricas, eclíptica y equinoccio de la fecha.
- `VSOP87E`: variables rectangulares baricéntricas, eclíptica y equinoccio J2000.

`RFC-0001` comparó las variantes y `ADR-0004` registró la decisión final:
`AST-0002` usará `VSOP87D`.

---

# 6. Precisión y Validación

La implementación usa los 1080 términos de longitud heliocéntrica terrestre
publicados en `VSOP87D.ear`:

- `L0`: 559 términos.
- `L1`: 341 términos.
- `L2`: 142 términos.
- `L3`: 22 términos.
- `L4`: 11 términos.
- `L5`: 5 términos.

Total: 1080 términos.

Los coeficientes fueron copiados con el mismo formato decimal visible en el
archivo oficial de IMCCE. No se truncaron manualmente ni se convirtieron a
`f32`.

Validación contra `vsop87.chk`:

- Casos validados: 10 fechas oficiales para la Tierra en `VSOP87D`.
- Tolerancia aceptada: `1.0e-9` rad.
- Error máximo observado: `3.968958495192964619e-11` rad.
- Error máximo observado: `2.274045708371511059e-9` grados.
- Error máximo observado: `8.186564550137440460e-6` segundos de arco.

La tolerancia `1.0e-9` rad se eligió porque `vsop87.chk` publica la longitud con
10 decimales en radianes. La tolerancia valida la implementación sin afirmar más
precisión que la publicada por la tabla de control.

# 7. Dependencias

AST-0002 depende de:

- Julian Day;
- tipos fuertes para ángulos (`Degrees`);
- coeficientes VSOP87D de longitud heliocéntrica terrestre.

AST-0002 no depende de:

- Luna;
- Tithi;
- Ekādaśī;
- reglas Gauḍīya.

---

# 8. API Pública

API implementada:

```rust
pub fn earth_heliocentric_longitude(jd: JulianDate) -> Degrees;
```

Internamente el cálculo se realiza en radianes y la conversión a grados ocurre
una sola vez, en el límite de la API pública.

La salida queda normalizada al intervalo:

```text
0° <= L < 360°
```

---

# 9. Fórmula Implementada

La fórmula implementada es:

```text
L(tau) = Σ tau^n * Σ A_i cos(B_i + C_i tau)
```

donde:

```text
tau = (JD - 2451545.0) / 365250.0
```

No se aplica división por `1e8` porque el archivo oficial `VSOP87D.ear` ya
publica las amplitudes angulares en radianes.

# 10. Benchmark

Benchmark implementado para:

- `earth_heliocentric_longitude`.

En modo debug/test se usa un número reducido de iteraciones para no convertir el
CI en una corrida larga. En release se mantiene una corrida de 10 millones de
iteraciones.

Resultado local de referencia:

```text
earth_heliocentric_longitude: 10000 iterations in 184.019667ms
```

Este valor es orientativo; no se considera aún una caracterización formal de
rendimiento.

---

# 11. Numerical Notes

- Se usa `f64` porque VSOP87D publica coeficientes decimales de alta precisión y
  Rust ofrece funciones trigonométricas estándar sobre `f64`.
- No se usa `f32`.
- No se usa SIMD.
- No se fuerza FMA.
- No se truncaron coeficientes para optimizar tamaño o velocidad.
- No se aplican optimizaciones prematuras sobre la evaluación de series.
- La normalización angular ocurre una sola vez, al final del cálculo en
  radianes.
- La conversión a `Degrees` ocurre una sola vez, en la API pública.
- El archivo de coeficientes permite explícitamente los lints de Clippy
  `excessive_precision` y `approx_constant` porque sus literales deben
  permanecer auditables contra IMCCE.
